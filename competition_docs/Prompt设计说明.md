# Prompt设计说明

## 一、设计目标

本项目的 Prompt 设计并不是单纯让模型“回答得更像人”，而是围绕反诈场景中的三个核心问题展开：

1. 如何让模型先看证据、再做判断，而不是直接生成主观结论。
2. 如何让不同模态分别提取稳定线索，再由主智能体统一决策。
3. 如何通过工具调用与结构化约束，降低大模型幻觉和结果漂移。

因此，本项目的 Prompt Engineering 目标可以概括为四点：

1. 角色明确：不同智能体只做自己负责的事情。
2. 过程可控：关键步骤必须经过工具链，不能跳步。
3. 输出可评测：关键结果必须结构化，便于批量评估。
4. 结论可追溯：风险等级、风险理由、下一步动作都要能回到具体证据。

除此之外，当前版本还特别强调一个新增目标：当证据质量不足、模态结果互相冲突或子智能体输出明显异常时，系统必须优先承认“不足以定性”，而不是为了给出完整结论强行补齐高风险判断。

## 二、Prompt总体架构

项目当前的 Prompt 体系分为六类：

1. 主智能体 Prompt：负责综合分析、工具编排和最终决策。
2. 图像子智能体 Prompt：负责图像内容取证。
3. 视频子智能体 Prompt：负责视频场景取证。
4. 音频子智能体 Prompt：负责语音话术与声纹风险取证。
5. 单图快速识别 Prompt：负责同步风险快判。
6. 聊天助手 Prompt：负责面向用户的连续对话式反诈咨询。

其中，真正承担案件判级责任的是主智能体；图像、视频、音频子智能体只提供模态级证据，不直接输出最终风险等级，不直接归档，不直接入库。

这种设计遵循“取证与裁决分离”的原则，避免单个模型同时承担全部职责导致的推理漂移，也有利于后续扩展更多模态和工具。

## 三、主智能体Prompt设计

### 3.1 角色定位

主智能体被定义为“多模态风控总分析专家”。它的核心职责不是生成一段自然语言描述，而是完成一次完整的反诈分析闭环，包括：

1. 聚合文本与多模态子结果。
2. 判断是否需要进一步检索信息。
3. 调用风险评估与动态分级工具。
4. 生成最终结构化报告。
5. 归档历史案件，并在必要时提交典型案例进入审核队列。

也就是说，主智能体是“总控与决策者”，而不是“任意发挥的聊天模型”。

### 3.2 阶段式任务拆解

主智能体 Prompt 采用显式阶段约束，把完整分析拆成六个阶段：

1. 信息收集与补全。
2. 当前案件风险评分。
3. 动态风险等级判定。
4. 最终结构化报告生成。
5. 典型案例增量沉淀。
6. 历史归档与结束。

这样设计的意义有两个：

1. 把“检索”“评分”“判级”“报告”“归档”从逻辑上拆开，减少模型跳步。
2. 强制模型先完成客观取证，再进入最终结论，提升稳定性和可解释性。

### 3.3 工具约束设计

主智能体 Prompt 中最关键的约束不是语言风格，而是工具调用顺序和终态约束。

当前主智能体可调用的关键工具包括：

1. `query_user_info`
2. `search_similar_cases`
3. `search_user_history`
4. `update_user_recent_tags`
5. `submit_current_risk_assessment`
6. `resolve_dynamic_risk_level`
7. `submit_final_report`
8. `write_user_history_case`
9. `upload_historical_case_to_vector_db`

Prompt 中对这些工具做了三类限制：

1. 顺序限制：必须先评分，再分级，再报告，再归档。
2. 权责限制：模型不能自行编造 `risk_score`、`dynamic_threshold`、`risk_level`，必须使用工具返回值。
3. 终态限制：只有“报告已提交 + 历史已归档”同时完成，任务才算结束。

这类设计直接把“高风险结论必须经过哪些步骤”写进了系统行为规则里，从 Prompt 层面抑制模型跳过关键流程。

### 3.4 风险判断约束

主智能体 Prompt 中还明确写入了几个关键判断原则：

1. 先围绕本次案件本身判断，不允许因为用户过去风险高，就直接把本次案件判高。
2. 历史分数只用于动态阈值，不直接替代本次案件证据。
3. 知识库命中和用户历史命中只描述相似性，不代表历史案件本身就直接等价于当前案件。
4. 若未命中，不代表现实中绝对没有先例，而是系统当前未检索到足以支撑判级的相似案例先验，因此不能脱离本案证据盲目抬高为高风险。
5. 高风险除了要求本次案件分数显著高于动态阈值外，还要求至少存在一个高风险相似命中作为先验支撑。
6. 中风险允许在存在高风险相似命中时进入阈值下方缓冲带，但动态阈值本身仍只由历史分数决定。

这些原则的作用是把“动态阈值”“相似命中先验”和“案件本体证据”分开，避免模型把“用户容易受骗”错误等价为“当前一定是诈骗”，也避免在缺少可验证先验时把案件直接判成高风险。

在最新版本中，主智能体 Prompt 又额外补强了两类边界：

1. 若图像、视频、音频子智能体结果明显错误、彼此冲突，或无法提供有效证据，主智能体必须明确承认证据不足。
2. 在没有文字、没有清晰语音、没有可核实客观线索的极端场景下，禁止给出任何高风险因素定论，禁止强行生成诈骗链路或诈骗关键词句。

这类约束的本质，是把“证据不足时如何收敛输出”也纳入 Prompt 设计，而不是只约束“证据充足时如何输出”。

## 四、子智能体Prompt设计

### 4.1 统一设计原则

图像、视频、音频三个子智能体 Prompt 虽然面向不同模态，但共享同一套设计思想：

1. 先判断内容性质，再提取客观信息。
2. 优先写“看到/听到什么”，再写“哪里可疑”。
3. 如果证据不足，明确说明“未发现明显可疑点”，避免过度推断。
4. 必须通过 `submit_analysis_result` 提交结构化结果。

这样做的目的是让子智能体尽量扮演“取证员”，而不是“法官”。

另外，子智能体当前还被加入了更严格的结构化输出约束：必须通过 `submit_analysis_result` 返回 `visual_impression`、`key_content`、`suspicious_points` 三个字段，且 `suspicious_points` 必须严格是 `[]string`。即使只有一条可疑点，也必须返回数组；若未发现异常，也必须返回如 `["未发现明显可疑点"]` 这样的标准化数组。这一设计直接降低了多模态链路里的解析歧义和格式漂移。

### 4.2 图像Prompt

图像 Prompt 重点约束四件事：

1. 区分真实拍摄、屏幕翻拍、数字合成、UI截图。
2. 扫描页面风格、排版和高风险视觉元素。
3. 尽可能完整提取图片中的文字、金额、网址、账号、按钮等客观信息。
4. 仅在有充分证据时列出可疑点，否则明确写“未发现可疑点”。

这种设计非常适合反诈场景中常见的聊天截图、转账页面、伪造通知、公文图片等高频素材。

### 4.3 视频Prompt

视频 Prompt 重点关注：

1. 视频是真实拍摄、录屏还是合成画面。
2. 是否存在动态诱导、虚假收益展示、伪装界面切换、字幕诱导等。
3. 是否出现网址、联系方式、倒计时、下载引导、转账引导等关键线索。

视频 Prompt 的重点不是“概括剧情”，而是把动态线索拆成可被主智能体复用的结构化证据。

需要说明的是，当前工程实现中，视频在进入模型前还会经过“媒体预处理层”做压缩、降帧和重编码，以满足多模态接口的请求体限制。Prompt 本身不负责压缩，但 Prompt 必须假设输入证据已经过规范化，并在此基础上只围绕可见客观内容做分析。这种“预处理负责可用性，Prompt 负责证据提取”的边界划分，有助于降低模块耦合。

### 4.4 音频Prompt

音频 Prompt 主要面向以下特征：

1. 声音是自然人声还是疑似合成语音。
2. 是否存在冒充客服、公检法、投资导师等典型脚本。
3. 是否有恐吓、催促、利益诱导、远控指导、验证码索取等高风险话术。
4. 背景音是否存在异常，如伪造办公环境、呼叫中心噪声等。

音频 Prompt 的输出同样强调“关键信息转录 + 可疑点提取”，而不是直接下结论。

## 五、快速识别与聊天Prompt设计

### 5.1 单图快速识别Prompt

单图快速识别是一个和主分析链路并行的轻量路径，其 Prompt 设计目标不是深度解释，而是：

1. 只输出 `高 / 中 / 低` 三档风险等级。
2. 给出一句简洁、客观、可追踪的理由。
3. 必须通过 `submit_image_quick_risk_result` 提交标准化结果。

这种 Prompt 设计适合用于“悬浮球快识别”“同步截图快判”等场景，强调时效性而不是完整闭环。

### 5.2 聊天助手Prompt

聊天助手 Prompt 的目标和主智能体不同，它主要解决“连续澄清与止损建议”问题。其设计重点包括：

1. 先识别高风险信号，再决定要不要追问。
2. 信息不足时优先提出 2-4 个关键澄清问题。
3. 始终给出明确风险分级与可执行建议。
4. 高风险时突出“先断联、先止损、再核验”。
5. 可结合用户画像与历史信息提供个性化建议。

聊天 Prompt 的作用是把复杂分析能力转化为用户可理解、可执行的对话体验。

在工程实现上，聊天助手并不是一次性输入、一次性回复，而是依托 Redis 会话上下文实现短时多轮连续对话。当前系统会按用户维度保存最近会话消息，存储内容包括用户输入、助手回复以及工具调用结果；会话键采用 `chat:context:<user_id>`，默认 TTL 为 5 分钟。用户持续发起对话时，系统会在读取历史后自动刷新 TTL；若用户希望重新开始新话题，则可以主动刷新上下文。这意味着聊天 Prompt 不只是“如何回答”，还实际依赖会话存储与上下文检索来保证追问、澄清和止损建议在多轮对话中保持一致。

## 六、少样本与结构化输出设计

虽然当前 Prompt 主体更强调规则约束和工具闭环，但从赛题材料表达角度，项目的 few-shot 设计思路同样清晰：

1. 黑样本示例：引导模型识别典型诈骗脚本、风险因子、关键证据。
2. 白样本示例：约束模型不要把正常银行、官方客服、正规售后、官方支付链路误判成诈骗。
3. 对照样本示例：同一场景下区分“正常引导”和“诈骗诱导”。

在输出层面，项目采用结构化字段约束，而不是只输出一段自由文本。关键输出字段包括：

1. 风险因子与评分结果。
2. 动态风险等级。
3. 风险理由。
4. 风险信号列表。
5. 下一步动作建议。
6. 可选的诈骗链路还原与诈骗关键词句。

这样做的好处是：

1. 便于前端可视化展示。
2. 便于批量评估和自动化测评。
3. 便于后续归档、检索与知识沉淀。

在当前版本里，这种结构化约束已经不仅体现在最终报告层，也体现在子智能体层。也就是说，结构化输出不再只是“最终给前端展示什么”，而是贯穿“子智能体取证 -> 主智能体聚合 -> 工具评分 -> 最终报告”整个链路的稳定性设计。

## 七、反幻觉与稳定性设计

本项目的 Prompt 设计并不是靠一句“请不要幻觉”来约束模型，而是通过多层机制共同控制：

1. 工具优先：关键结论必须走工具，不允许模型自行编造核心状态。
2. 证据优先：结论必须围绕风险信号、相似命中、评分结果和动态阈值解释。
3. 阶段限制：未完成评分与分级前，不允许直接提交最终报告。
4. 角色分离：子智能体负责取证，主智能体负责决策，减少单模型混用角色。
5. 格式收敛：子智能体与主智能体关键输出都被限制在工具 schema 内，降低字段漂移。
6. 证据不足收敛：当模态结果不可靠或客观证据不足时，系统必须收敛到“证据不足”的谨慎输出，而不是升级风险判断。
7. 终态控制：未完成归档时任务不结束，避免出现“回答了但系统没落库”的假完成状态。

这些机制共同保证了系统输出不仅“像正确答案”，而且“流程上真的经过了可校验步骤”。

## 八、与赛题要求的对应关系

本项目的 Prompt 设计与赛题要求具有直接对应关系：

1. 对应“Prompt Engineering 设计高效系统提示词”：主智能体、子智能体、快速识别、聊天助手均有独立提示词。
2. 对应“知识库检索避免幻觉”：Prompt 明确要求优先检索知识库和用户历史，并用命中结果支撑判断。
3. 对应“多模态融合判别”：子智能体分模态取证，主智能体再统一聚合。
4. 对应“个性化风险评估”：Prompt 明确把 `historical_score`、动态阈值和相似命中纳入判级链路。
5. 对应“报告生成与全链路闭环”：Prompt 中强制要求提交结构化报告并完成历史归档。

因此，Prompt 在本项目中不是附属配置，而是驱动多智能体协同、风险控制与系统闭环的核心设计之一。

## 九、结论

本项目的 Prompt 设计可以概括为一句话：

不是让模型“自由发挥地判断诈骗”，而是通过角色分工、阶段约束、工具闭环、结构化输出和证据优先策略，把大模型能力收束进一个可控、可评测、可复核的反诈智能体系统。

这也是本项目在比赛材料中值得重点强调的一个技术亮点：Prompt 设计不是装饰层，而是系统稳定性、可解释性和落地能力的重要基础。
