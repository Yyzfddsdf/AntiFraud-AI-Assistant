# 部署与演示指南

## 一、文档目的

本文档用于说明本项目的部署方式、运行依赖、核心配置和现场演示流程，目标是让评委、指导老师或团队成员能够在较短时间内完成：

1. 服务启动。
2. 账号准备。
3. 多模态分析闭环演示。
4. 实时预警与家庭联防演示。
5. 后台案件知识库运营演示。

本文档以当前仓库实际实现为准，不额外依赖未提交的脚本或外部工程。

## 二、运行环境

### 2.1 基础环境

建议环境如下：

1. Windows 10/11 或其他可运行 Go 的环境。
2. Go 1.25。
3. Redis 6.x 及以上。
4. 可访问外部模型服务或兼容 OpenAI 协议的模型网关。
5. 本地可写目录，用于 SQLite 数据库文件与缓存文件。

### 2.2 当前主要依赖

项目当前使用：

1. Gin 作为 Web 框架。
2. SQLite 作为主业务库和历史案件库。
3. Redis 作为统一缓存层。
4. OpenAI 兼容协议模型接口，用于主智能体、子智能体、聊天助手和 embedding。

## 三、关键文件

部署时主要关注以下文件：

1. `cmd/api/main.go`：服务启动入口。
2. `internal/platform/config/config.json`：主配置文件。
3. `README.md`：开发者侧说明。
4. `API.md`：接口文档。
5. `DB/`：数据库默认目录。

补充说明：项目后端当前采用标准化六边形结构，部署时可以按以下理解代码组织：

1. `cmd/`：启动入口。
2. `internal/bootstrap/`：组合根与路由装配。
3. `internal/platform/`：配置、数据库、缓存、模型与搜索等基础设施。
4. `internal/modules/`：按业务域拆分的 `domain / application / adapters` 模块。
5. `web/`：桌面端与移动端静态页面资源。

比赛材料相关文档位于：

1. `competition_docs/大众反诈系统项目文档_获奖导向版.md`
2. `competition_docs/项目概要介绍_获奖导向版.md`
3. `competition_docs/答辩PPT文案_获奖导向版.md`
4. `competition_docs/Prompt设计说明.md`

## 四、配置说明

### 4.1 配置读取方式

项目采用“配置文件 + 环境变量”混合配置方式：

1. 普通运行参数主要通过环境变量控制。
2. 模型参数、Prompt、Redis、搜索与重试参数主要通过 `internal/platform/config/config.json` 控制。
3. API Key 类字段支持环境变量覆盖 `internal/platform/config/config.json`。

### 4.2 必要环境变量

建议部署前至少设置以下环境变量：

1. `PORT`：服务端口，默认 `8081`。
2. `JWT_SECRET`：JWT 密钥，生产环境必须设置。
3. `INVITE_CODE_ADMIN`：管理员升级邀请码。
4. `DB_PATH`：主业务数据库路径，默认 `DB/auth_system.db`。
5. `HISTORICAL_CASE_DB_PATH`：历史案件库路径，默认 `DB/historical_case_library.db`。

### 4.3 API Key 环境变量覆盖

以下字段若对应环境变量存在且非空，会优先覆盖 `internal/platform/config/config.json` 中的 `api_key`：

1. `AGENT_MAIN_API_KEY`
2. `AGENT_IMAGE_API_KEY`
3. `AGENT_IMAGE_QUICK_API_KEY`
4. `AGENT_VIDEO_API_KEY`
5. `AGENT_AUDIO_API_KEY`
6. `AGENT_CASE_COLLECTION_API_KEY`
7. `EMBEDDING_API_KEY`
8. `CHAT_API_KEY`
9. `ADMIN_CHAT_API_KEY`
10. `TAVILY_API_KEY`

### 4.4 `internal/platform/config/config.json` 重点检查项

启动前建议重点核对以下节点：

1. `agents.main / image / image_quick / video / audio / case_collection`
2. `embedding`
3. `chat`
4. `admin_chat`
5. `redis`
6. `prompts.main / image / image_quick / video / audio / case_collection`
7. `alert_ws`
8. `family_alert_ws`

如果当前环境不具备某类模型能力，请优先保证主智能体、图像快速识别和 embedding 至少可用。

## 五、启动步骤

### 5.1 安装依赖

在项目根目录执行：

```bash
go mod tidy
```

### 5.2 确认 Redis 可用

项目中的验证码、限流、聊天上下文、向量缓存均依赖 Redis。  
请确认 `internal/platform/config/config.json` 中的 `redis.addr` 可达，并且对应 Redis 服务已启动。

补充说明：本项目对 Redis 缓存已做工程化生命周期管理，演示时一般不需要人工清空 Redis。聊天上下文支持通过业务接口主动刷新，统计与地图类缓存则通过 TTL 与版本号机制自然失效，避免现场演示过度依赖手工运维操作。

### 5.3 启动服务

在项目根目录执行：

```bash
go run ./cmd/api
```

若启动成功，默认可访问：

1. API 基地址：`http://localhost:8081/api`
2. Web 页面：`http://localhost:8081/`
3. 独立前端请分别通过各自开发地址访问，并调用 `http://localhost:8081/api` 完成联调
4. 移动端页面：`http://localhost:8081/mobile`

### 5.4 启动成功的判断方式

满足以下条件即可视为服务已正常启动：

1. 终端未出现 `load app config failed`、`init persistence failed` 等致命错误。
2. 使用独立前端访问登录页，并确认前端能够正常请求 `/api`。
3. `GET /api/auth/captcha` 可正常返回验证码。

## 六、初始化与账号准备

### 6.1 注册普通用户

建议先通过以下流程创建一个普通用户：

1. 调用 `GET /api/auth/captcha` 获取图形验证码。
2. 调用 `POST /api/auth/sms-code` 获取短信验证码。
3. 调用 `POST /api/auth/register` 完成注册。
4. 调用 `POST /api/auth/login` 获取 JWT。

当前演示环境中，短信验证码默认固定为：

```text
000000
```

### 6.2 创建管理员账号

若已有普通用户账号，可通过：

```text
POST /api/upgrade
```

使用 `INVITE_CODE_ADMIN` 对应的邀请码升级为管理员。

建议准备两类账号：

1. 普通用户：用于多模态分析、聊天、家庭中心、风险总览。
2. 管理员用户：用于案件审核、历史案件库管理、案件采集和后台统计。

### 6.3 准备家庭联防账号

若需要演示家庭联防，建议至少准备两个普通用户：

1. 被守护用户：用于产生高风险案件。
2. 守护人用户：用于接收家庭风险通知。

建议先完成以下准备：

1. 创建家庭组。
2. 邀请守护人加入家庭。
3. 配置守护关系。
4. 建立家庭通知 WebSocket 连接。

## 七、推荐演示顺序

为了保证现场演示节奏稳定，建议采用“由浅入深、由快到全”的顺序：

1. 单图快速识别。
2. 多模态完整分析。
3. 用户风险历史与总览。
4. 实时预警推送。
5. 家庭联防通知。
6. 管理员案件审核与知识库运营。

## 八、演示闭环说明

### 8.1 演示一：单图快速识别

目标：证明系统具备低延迟单图快判能力。

建议步骤：

1. 登录普通用户，获取 JWT。
2. 调用 `POST /api/scam/image/quick-analyze`。
3. 传入一张聊天截图、伪造页面截图或收款页面截图。
4. 展示同步返回的 `risk_level` 与 `reason`。

建议说明重点：

1. 该接口不进入任务队列，适合移动端主动快识别。
2. 适合悬浮球、截图快判、页面风险初筛场景。

### 8.2 演示二：多模态完整分析

目标：展示“输入 - 识别 - 报告 - 归档”的主链路。

建议步骤：

1. 调用 `POST /api/scam/multimodal/analyze`。
2. 输入文本，并可附带图片、音频或视频 Base64 内容。
3. 获得 `task_id` 后轮询：
   - `GET /api/scam/multimodal/tasks`
   - `GET /api/scam/multimodal/tasks/:taskId`
4. 展示最终报告中的：
   - 风险等级
   - 诈骗类型
   - 风险理由
   - 下一步动作
   - 可选诈骗链路还原

建议说明重点：

1. 子智能体负责取证，主智能体负责最终决策。
2. 风险评分、动态分级、报告提交、历史归档均通过工具闭环完成。

### 8.3 演示三：风险历史与总览

目标：展示系统不是“一次性判断工具”，而是可持续积累的反诈系统。

建议步骤：

1. 在普通用户侧调用：
   - `GET /api/scam/multimodal/history`
   - `GET /api/scam/multimodal/history/overview`
2. 展示：
   - 历史案件记录
   - 高/中/低风险统计
   - 风险趋势分析

建议说明重点：

1. 系统具备持续记忆能力。
2. 用户历史风险会参与动态阈值判级。

### 8.3.1 演示三补充：聊天上下文连续性

目标：展示系统能够记住上一轮澄清结果，而不是每轮都从零开始回答。

建议步骤：

1. 先调用 `POST /api/chat` 提出一个信息不完整的可疑场景。
2. 继续调用 `POST /api/chat` 补充“是否已转账”“对方身份说法”“是否索要验证码”等关键信息。
3. 调用 `GET /api/chat/context`，展示当前 Redis 会话上下文中已经保存的消息与剩余 TTL。
4. 调用 `POST /api/chat/refresh` 清空上下文。
5. 再次调用 `GET /api/chat/context`，展示上下文已被清空。

建议说明重点：

1. 当前聊天上下文按用户维度存储在 Redis 中，键为 `chat:context:<user_id>`。
2. 默认 TTL 为 5 分钟，活跃会话会自动续期。
3. 存储内容不仅包括用户消息和助手消息，也包括工具调用结果，因此模型能够在下一轮对话中复用前一轮检索与澄清信息。

### 8.4 演示四：实时预警

目标：展示系统能够主动推送风险事件，而不是只被动回答。

建议步骤：

1. 建立普通用户的 WebSocket 连接：
   - `GET /api/alert/ws?token=<JWT_TOKEN>`
2. 提交一个中风险或高风险案件。
3. 展示服务端在告警窗口内主动推送 `risk_alert` 消息。

建议说明重点：

1. 告警不是前端自己轮询拼出来的，而是后端基于历史归档事件主动推送。
2. 这体现了系统从“查询型工具”向“主动守护系统”的升级。

### 8.5 演示五：家庭联防

目标：展示“个人识别 -> 家庭通知”的联动闭环。

建议步骤：

1. 提前配置家庭组与守护关系。
2. 被守护用户提交一个高风险案件。
3. 守护人侧建立家庭通知 WebSocket：
   - `GET /api/families/notifications/ws?token=<JWT_TOKEN>`
4. 展示守护人接收到家庭高风险通知。

建议说明重点：

1. 家庭系统与主分析链路保持低耦合。
2. 只有高风险案件才触发家庭联防通知，避免过度打扰。

### 8.6 演示六：管理员知识库运营

目标：展示系统具备知识治理和持续进化能力。

建议步骤：

1. 使用管理员账号查看待审核案件：
   - `GET /api/scam/review/cases`
2. 查看详情：
   - `GET /api/scam/review/cases/:recordId`
3. 审核通过：
   - `POST /api/scam/review/cases/:recordId/approve`
4. 查看历史案件库与全国态势地图：
   - `GET /api/scam/case-library/cases`
   - `GET /api/scam/case-library/cases/overview`
   - `GET /api/scam/case-library/cases/graph`
   - `GET /api/scam/case-library/cases/geo-map` (全国反诈态势地图数据)

建议说明重点：

1. 典型案件不会直接入正式知识库，而是先进入待审核队列。
2. 系统具备知识沉淀、统计总览、图谱分析和全国多级地理态势展示能力。
3. 地理态势地图支持从全国到省、市、县区级的多级下钻，实时反映区域风险。

## 十、常见问题排查

### 10.1 启动时报配置文件错误

常见现象：

1. `load app config failed`
2. `failed to decode config file`
3. `api_key is required`

排查建议：

1. 确认 `internal/platform/config/config.json` 是有效 UTF-8 JSON。
2. 确认必填模型参数完整。
3. 若不想把 API Key 写入配置文件，可改用环境变量覆盖。

### 10.2 Redis 连接失败

常见现象：

1. 验证码、限流、聊天上下文异常。
2. 向量缓存预热失败。

排查建议：

1. 确认 Redis 已启动。
2. 确认 `internal/platform/config/config.json -> redis.addr` 正确。
3. 确认本机防火墙未拦截目标端口。

### 10.3 模型调用失败

常见现象：

1. 上游模型接口报错。
2. 多模态任务卡在处理中。
3. 快速识别接口返回 `502`。

排查建议：

1. 检查 `base_url`、`model`、`api_key`。
2. 检查当前 provider 是否支持对应模态。
3. 优先保证 `agents.main`、`agents.image_quick`、`embedding` 可用。

### 10.4 WebSocket 无消息

排查建议：

1. 确认传入的是合法 JWT。
2. 确认在告警窗口内确实产生了中高风险或高风险案件。
3. 确认家庭守护关系已经提前建立完成。
