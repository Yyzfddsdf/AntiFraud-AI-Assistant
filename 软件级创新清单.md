# 软件级创新清单

本文档收缩为最小可落地版本，只保留一个方向：

悬浮气泡主动触发 -> 截图 -> OCR + 短判断 -> 高风险进入主分析 -> 可选通知家人

目标是降低软件复杂度，不做后台实时监控，不做持续截屏，不做重型设备风控平台。

## 1. 最终方案定义

### 1.1 产品定义

一个显眼的悬浮气泡，用户在聊天、网页、转账、下载页面觉得可疑时，主动点一下。

系统执行以下动作：

1. 截取当前画面
2. 提取 OCR 文本
3. 做一次轻量短判断
4. 给出三档结果：
   - 安全
   - 可疑，建议主分析
   - 高风险，立即主分析
5. 用户确认后进入现有主分析流程
6. 必要时通知家庭成员

### 1.2 明确不做的内容

1. 不做后台实时监控
2. 不做持续截屏
3. 不做复杂设备心跳体系
4. 不做全量规则中心
5. 不做一堆新的风控表
6. 不做自动强制跳转
7. 不做系统级硬阻断

## 2. 软件端只做什么

### 2.1 必做功能

1. 悬浮气泡
2. 点击后截图
3. OCR 提取文字
4. 本地硬规则粗判
5. 请求后端做轻量短判断
6. 弹出短结果卡片
7. 用户点击后进入主分析

### 2.2 交互流程

1. 用户点击悬浮气泡
2. App 截取当前画面
3. 端上先做 OCR 和关键词规则判断
4. 如果明显安全，直接返回“基本安全”
5. 如果命中可疑规则，再请求后端短判断
6. 后端返回：
   - 安全
   - 可疑
   - 高风险
7. 如果是可疑或高风险，用户可点“立即主分析”
8. 如果用户点“通知家人”，再走家庭联防

### 2.3 软件端模块

只保留四块：

1. `FloatingBubbleService`
2. `ScreenCaptureService`
3. `QuickCheckClient`
4. `MainAnalysisLauncher`

## 3. 后端只新增什么

### 3.1 复用现有接口

以下接口直接复用：

1. `POST /api/scam/multimodal/analyze`
2. `GET /api/scam/multimodal/history`
3. `GET /api/scam/multimodal/history/overview`
4. `POST /api/families`
5. `GET /api/families/me`
6. `POST /api/families/invitations`
7. `POST /api/families/invitations/accept`
8. `POST /api/families/guardian-links`
9. `GET /api/families/notifications/ws`

### 3.2 新增一个轻接口

只新增一个核心接口：

`POST /api/device-risk/quick-checks/image`

用途：

1. 接收截图
2. 接收 OCR 文本
3. 做轻量短判断
4. 返回是否建议进入主分析

### 3.3 可选新增一个桥接接口

如果要把“通知家人”做干净，再新增一个桥接接口：

`POST /api/device-risk/escalations`

用途：

1. 接收高风险升级请求
2. 服务端内部桥接到现有 `family_system`
3. 不让客户端直接操作家庭通知逻辑

## 4. 轻接口设计

### 4.1 请求

`POST /api/device-risk/quick-checks/image`

```json
{
  "source_app": "com.tencent.mm",
  "image_base64": "base64-image",
  "ocr_text": "客服说我开通了会员需要退款，需要验证码",
  "trigger_mode": "floating_bubble"
}
```

### 4.2 返回

```json
{
  "risk_level": "high",
  "risk_score": 86,
  "summary": "检测到退款诈骗和验证码诱导特征",
  "should_open_main_analysis": true
}
```

### 4.3 返回字段说明

1. `risk_level`
   - `safe`
   - `suspicious`
   - `high`
2. `risk_score`
   - 0 到 100
3. `summary`
   - 一句话短结论
4. `should_open_main_analysis`
   - 是否建议进入现有主分析流程

## 5. 主分析怎么接

### 5.1 进入条件

只有以下情况才进入主分析：

1. 后端返回 `high`
2. 后端返回 `suspicious` 且用户主动点击“立即主分析”

### 5.2 主分析复用现有能力

进入主分析后，继续复用：

`POST /api/scam/multimodal/analyze`

这样主分析仍然是当前项目已有主链路，不需要重做。

### 5.3 家庭联防复用现有能力

如果主分析结果为高风险，可继续复用：

1. 现有家庭关系
2. 现有守护关系
3. 现有家庭通知 WebSocket

## 6. 数据库怎么压缩

### 6.1 最省方案

先不新增表。

短判断只返回结果，不落库。
只有用户进入主分析后，才写现有历史记录。

### 6.2 如果必须留痕

最多新增一张表：

`device_risk_quick_checks`

建议字段：

1. `id`
2. `user_id`
3. `source_app`
4. `risk_level`
5. `risk_score`
6. `summary`
7. `ocr_text`
8. `created_at`

这样就够了，不再拆设备表、规则表、动作表、情报表。

## 7. 最小开发清单

### 7.1 Android 端

1. 悬浮气泡
2. 截图能力
3. OCR
4. 调用 `/api/device-risk/quick-checks/image`
5. 风险结果卡片
6. 跳转主分析页

### 7.2 后端

1. 新增 `/api/device-risk/quick-checks/image`
2. 内部复用现有多模态能力做轻量判断
3. 可选新增 `/api/device-risk/escalations`

### 7.3 比赛/演示重点

只展示三段：

1. 点悬浮气泡
2. 秒级返回风险短判断
3. 一键进入主分析并通知家人

## 8. 为什么这个版本更合适

1. 比后台实时监控简单很多
2. 比全量设备风控平台便宜很多
3. 比持续 OCR 和持续截图更像正常产品
4. 能最大化复用你当前项目
5. 容易做出演示效果

## 9. 设计约束

### 9.1 UTF-8

文档、OCR 文本、返回摘要统一使用 UTF-8。

### 9.2 低耦合

只新增轻接口，不侵入现有 `multimodal` 和 `family_system` 表结构。

### 9.3 防连环报错

短判断失败时，不影响现有主分析；家庭通知失败时，不影响短判断结果返回。
